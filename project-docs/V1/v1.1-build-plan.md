# Build Plan: Ollama Terminal Chat Interface v1.1

Source PRD: `project-docs/v1.1-prd.md`

## Goal

Revise the current v1 terminal interface from a plain line-based REPL to an `ink` TUI with a persistent bordered output box, persistent bordered input box, and status line showing the active model and approximate context usage.

The functional scope stays the same as v1. This plan should preserve the existing Ollama client, conversation model, storage repository, search behavior, and command semantics wherever possible.

## Current Baseline

Already implemented in v1:

- TypeScript + Node.js npm CLI.
- `hearth` bin entry.
- Ollama `/api/tags` model listing and `/api/chat` streaming.
- Flat JSON conversation storage with atomic writes.
- Conversation lifecycle helpers.
- Slash command parsing.
- Search.
- Approximate context usage.
- Markdown-ish terminal rendering and syntax highlighting.
- Tests for storage, commands, search, conversation state, and stream parsing.

Primary change for v1.1:

- Replace `src/cli/repl.ts` with an Ink app and supporting UI components.
- Keep `src/core`, `src/ollama`, `src/storage`, and `src/search` as stable service layers, changing them only where the UI needs better event/state hooks.

## Recommended Dependency Changes

Add:

- `ink`
- `react`

Layout rules:

- Output box consumes remaining height.
- Input box uses fixed height.
- Status line renders directly below the input box.
- Borders use the same style across input and output.
- Empty input shows dim placeholder text.
- The active model updates immediately after `/model`.
- Context usage updates after state changes and after completed responses.

## State Model

The TUI should own display state, while `ChatApp` remains the domain/service layer.

Suggested UI state:

```ts
type TuiState = {
  input: string;
  messages: DisplayMessage[];
  activeConversationId?: string;
  activeModel?: string;
  systemPrompt?: string;
  contextEstimate?: ContextEstimate;
  isStreaming: boolean;
  notice?: string;
  error?: string;
};
```

`DisplayMessage` can initially mirror saved messages:

```ts
type DisplayMessage = {
  id: string;
  role: "user" | "assistant" | "system";
  content: string;
  isStreaming?: boolean;
};
```

Important detail:

- During streaming, append a temporary assistant display message.
- Batch incoming deltas into that message.
- When the stream completes, refresh display messages from `ChatApp.currentConversation` so UI state and persisted state are aligned.

## Streaming Strategy

Do not call React state updates for every token.

Use a batching helper:

- Collect streamed deltas in a mutable buffer.
- Flush to Ink state every 30-60 ms, or immediately at stream completion.
- Keep the output box pinned to the bottom while streaming.

This directly addresses the v1.1 PRD risk around Ink re-render cost.

## Command Handling

Keep the existing slash commands and command parser:

- `/help`
- `/models`
- `/new [model]`
- `/list`
- `/load <id>`
- `/save`
- `/model <name>`
- `/system [prompt]`
- `/system clear`
- `/search <query>`
- `/regen`
- `/edit`
- `/clear`
- `/exit`

Change command output from printed terminal text to display messages/notices inside the output box.

Examples:

- `/list` renders a system-style result message containing saved conversations.
- `/search query` renders matching snippets.
- `/models` renders installed model names.
- `/help` renders command help.
- Errors render as a visible error notice inside or just above the status line.

## Implementation Notes

Preserve the core/service boundary:

- `ChatApp` should remain usable without Ink.
- The TUI should not directly read/write conversation files.
- The TUI should not call Ollama directly.
- All persistence should still flow through `ConversationRepository`.

Likely `ChatApp` adjustments:

- Add a way to expose current messages safely for UI refresh.
- Consider splitting `submitUserMessage` into lower-level operations if streaming UI needs finer control.
- Preserve existing tests while adding focused tests for any new behavior.

## Risks and Mitigations

**Risk: Streaming re-renders are too frequent.**

Mitigation: batch deltas every 30-60 ms and flush once at completion.

**Risk: Markdown rendering is awkward in Ink.**

Mitigation: first ship readable Markdown inside boxes, then improve styling. Do not block the TUI migration on perfect Markdown parity.

**Risk: Resize behavior is visually glitchy.**

Mitigation: explicitly test common terminal sizes and avoid hard-coded dimensions except for input/status height.

**Risk: `/edit` is clumsy in a controlled input.**

Mitigation: implement a clear edit mode with prefilled input rather than opening a separate prompt.

## Definition of Done for v1.1

- The plain scrolling REPL is replaced by an Ink TUI.
- The screen has a persistent bordered output box and bordered input box.
- The active model and context estimate appear in a persistent status line.
- Streaming responses render inside the output box.
- Existing v1 commands still work.
- Conversation persistence and storage format remain backward compatible.
- Resize behavior is manually verified.
- Build, typecheck, and tests pass.
