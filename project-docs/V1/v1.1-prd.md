## 1. What Changed From v1

v1 left the interface style as an open decision: a plain REPL was recommended as the starting point, with a richer TUI (`ink`) flagged as something to revisit if the UX felt limited.

That decision is now made. Based on a reference UI (OpenCode's terminal interface — see below), this PRD locks in a TUI built with `ink`, specifically a layout with a **bordered output box** and a **bordered input box**, replacing the plain scrolling REPL.

Everything else from v1 — all ten user stories, all acceptance criteria, the data storage format, the Ollama API integration, the testing strategy — is unchanged. This document only revises the **Architecture Overview** and **Non-Goals** sections of v1, and adds new UI-specific acceptance criteria. It does not re-litigate scope.

---

## 2. Reference

The visual reference is OpenCode's terminal welcome/chat screen: a bordered input region anchored near the bottom of the terminal, with placeholder prompt text inside it, a status line directly beneath showing the active mode and model (e.g., `Build · qwen3:0.6b Ollama`), and a row of keybinding hints to the side.

**What's adopted from this reference:**
- A persistent, anchored bordered input box (not a plain `> ` prompt scrolling with the rest of the output).
- A persistent, anchored bordered output box where streamed responses render, scrollable as conversation grows.
- A status line under the input showing the active model.
- Both boxes redraw cleanly in place as state changes, rather than the terminal scrolling raw text upward.

**What's explicitly not adopted:**
- Agent/mode switching (OpenCode's `Build`/`Plan` modes via Tab) — this implies functionality outside Tier 1/2 scope (no agentic modes exist in this tool).
- The `ctrl+p` command palette — not required; existing slash commands (`/new`, `/load`, `/model`, etc.) remain text-typed, not palette-driven, unless explicitly scoped later.
- The large block-letter ASCII logo — cosmetic, optional, not a functional requirement. Can be added later without affecting architecture.
- Session sharing, tabs, or multi-window chrome — not part of this tool's model at all.

## 3. User Experience & Functionality (Additions to v1)

### New User Stories

11. As a user, I want the input area to stay visually anchored at the bottom of the terminal, so that I always know where to type regardless of how much output has scrolled by.
12. As a user, I want to see the active model name in a persistent status line, so that I don't have to run a separate command to check what I'm currently talking to.
13. As a user, I want the output region to scroll independently of the input box, so that reviewing past responses doesn't disrupt my ability to type a new message.

### New Acceptance Criteria

**Layout**
- The terminal screen is divided into two persistent regions: an output box (top, scrollable) and an input box (bottom, fixed height).
- Both boxes have a visible border. Border style is consistent across both (e.g., both rounded, or both single-line — not mixed).
- The input box displays placeholder text (e.g., `Ask anything...`) when empty, styled distinctly (dimmed) from actual typed input.

**Status line**
- A status line renders directly beneath the input box, showing at minimum the active model name.
- The status line updates immediately when the model is switched via `/model`, without requiring a manual screen refresh.
- The context usage indicator (already required in v1, Tier 2) is displayed as part of this status line rather than as a separate printed message.

**Streaming inside the output box**
- Streamed response tokens render inside the output box as they arrive, with the box auto-scrolling to keep the latest content visible.
- Markdown and syntax highlighting (already required in v1) render correctly inside the bordered output box, not just in raw terminal output.

**Resize handling**
- Both boxes resize correctly if the terminal window is resized mid-session, without corrupting the border rendering or losing scroll position unreasonably.

**Non-Goals (additions to v1's list)**
- No agent-mode switching (Build/Plan or equivalent) — not part of this tool's functionality.
- No command palette (`ctrl+p` style) — slash commands remain text-typed.
- No ASCII-art logo requirement — purely optional polish, not an acceptance criterion.
- No tab/window chrome, session sharing, or multi-pane layouts beyond the two boxes described above.

## 4. Technical Specifications (Revisions to v1)

### Architecture Overview — Revised

- **Language/runtime:** TypeScript on Node.js. Unchanged.
- **Package management:** npm. Unchanged.
- **Interface style:** **Decided.** Built with `ink` (React-style terminal rendering), not a plain REPL loop. This reverses v1's "start with REPL" recommendation — the bordered, persistently anchored two-box layout requires component-based re-rendering that a line-based REPL cannot reasonably support.
- **Rationale for this reversal:** v1 treated REPL-vs-TUI as low-risk to defer. Once a concrete bordered-box layout with a persistent status line became a requirement, that deferral was no longer tenable — these are layout primitives `ink` provides natively (`<Box>` with `borderStyle`) and a REPL does not.
- **UI composition:**
  - Root layout: a vertical flex container with two children — output box (flexible height, scrollable) and input box (fixed height).
  - Output box: renders the conversation as a list of message components, each handling its own Markdown/syntax-highlighting rendering.
  - Input box: a controlled text input component, with the status line rendered as a sibling element directly below it, inside the same bordered container or immediately adjacent (visual decision to make during implementation — both are reasonable; consistency with the OpenCode reference favors immediately adjacent, just below the border).
- **Data flow:** Unchanged from v1 — only the rendering layer changes, not how messages are sent to or received from Ollama's `/api/chat`.

### New Risk: ink-specific implementation considerations

- **Streaming + component re-render cost:** `ink` re-renders on state change. Streaming tokens one-by-one into a React-style component tree at high frequency can cause performance issues if not throttled or batched. Needs a buffering/batching strategy (e.g., update render state every N milliseconds or N tokens, not on every single token) rather than naive per-token state updates.
- **Terminal resize handling:** `ink` supports resize events, but the box layout needs explicit handling to avoid border corruption or scroll-position loss on resize — this should be tested explicitly, not assumed to work by default.
- **Library maturity gap vs. the reference:** OpenCode's polish comes from Go's Bubble Tea library, which is generally considered more mature for this exact use case than `ink`. The bordered-box layout itself is fully achievable in `ink`, but matching OpenCode's exact level of visual polish may take more iteration in `ink` than it would in Bubble Tea. This is an accepted tradeoff given the TypeScript/Node.js stack decision already locked in the v1 PRD — not a reason to revisit the language choice.
